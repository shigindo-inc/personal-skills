# Dashboard Update — `36_応募ダッシュボード.md` 編集規約

`30_応募活動/36_応募ダッシュボード.md` のセクション 1（企業別進捗テーブル）への追記ルール。

## 追記対象セクション

**セクション 1 のみ**を編集する。セクション 2（全体サマリ）・3（内定比較）・4（SNS スカウト）は触らない。

## テーブル列の対応

`36_応募ダッシュボード.md` のテーブルヘッダ：

```
| # | 企業名 | フォルダ | エージェント経由 | 応募日 | 現フェーズ | 次アクション | 期日 | 評価スコア | 職種 / 年収提示 | メモ |
```

company-prep が企業新規登録時に埋める値：

| 列 | 値 |
|----|---|
| # | 既存最大番号 + 1（既存テーブルから採番） |
| 企業名 | ユーザー指定の社名 |
| フォルダ | `[companies/{企業名}/](../20_面接対策/companies/{企業名}/README.md)` |
| エージェント経由 | `--agent <名>` 指定時はその名、無ければ `直応募` |
| 応募日 | 未応募時は `—`、応募確定時は YYYY/MM/DD |
| 現フェーズ | `情報収集中` 固定 |
| 次アクション | `企業情報埋め完了次第、評価判定` or `応募軸確定` |
| 期日 | `—`（情報収集中は期日無し） |
| 評価スコア | `必須仮 X / 5（面談で確定）/ 希望 — / 21` |
| 職種 / 年収提示 | JD 抽出値 or `要確認` |
| メモ | 業界 + 規模 + 注目フラグ 1〜2 個（例：`AI スタートアップ / 50 名 / CTO 距離近そう`） |

## 編集手順

1. **既存テーブル末尾の番号確認**：Grep `^| [0-9]+ |` で最大番号を取得
2. **記入例行（`> **記入例**：` 以下）を維持**：上書きしない
3. **空行（`| 1 | — | — | ...`）があれば、最初の空行を上書き**。空行が無ければテーブル末尾に追加
4. **テーブルカラム数（パイプ数）を厳格に維持**：他行のパイプ数を Grep で確認してから挿入

## セクション 2 の自動再集計（行わない）

セクション 2「全体進捗サマリ」の件数（情報収集中 / 応募済 / 面接フェーズ / 内定 / 辞退・不合格）は **company-prep では更新しない**。週次の手動レビュー or 将来の専用 skill で実施。

新企業を追加した場合、セクション 1 のフェーズ列だけが更新され、セクション 2 とのずれは「次回サマリ更新時に同期」というルール。

## `最終更新` の扱い

`36_応募ダッシュボード.md` の冒頭 `**最終更新**：—` は **更新する**（当日 YYYY/MM/DD）。

## エージェント経由応募時の追加処理

`--agent <名>` が指定された場合、`30_応募活動/agents/{名}.md` の存在を確認し：
- 存在する → そのファイルの「紹介済企業」セクション（あれば）に企業名を追記
- 存在しない → ユーザーに「`agents/{名}.md` が見つからない。`30_応募活動/agents/_TEMPLATE.md` から複製するか確認」と報告（自動生成はしない、`agent-interview-prep` skill の責務）

## 検証

- セクション 1 のテーブルが Markdown として valid（パイプ数一致、ヘッダ行 + セパレータ行 + データ行）
- 新規追加した行のリンク `[companies/{企業名}/](...)` が実在パスを指す
- 番号が連続している（飛び番号が無い）
