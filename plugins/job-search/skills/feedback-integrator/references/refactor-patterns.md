# Refactor Patterns — リファクタの典型パターン集

`impact-matrix.md` で特定した影響ファイルに対して、どう編集するかの典型パターン。「追記」「章再構成」「写像更新」「優先度入替」の 4 系統。

## P1：章本文への追記（最も軽微）

**該当 FB 例**：「B-4 に Azure と OpenAI の使い分け説明が足りない」

**手順**：
1. 該当章の既存本文を Read
2. 不足箇所を特定（既存本文の流れを壊さない位置）
3. 追記内容を 30〜60 秒で話せる長さに収める
4. `**深掘り想定**：` 節があれば、想定追加質問も併記

**注意**：
- 既存の論理流れを壊さない
- 24 D-1 ルール：検証不能な数値を新規投入しない
- 「面接官が突きそうな反問」を `**深掘り想定**` に追加

## P2：章本文の再構成（中規模）

**該当 FB 例**：「B-2 学歴ギャップ / 在籍年数の説明、30 秒版・時系列版・深掘り 12 パターンが整理されてない」

**手順**：
1. 既存章の全体構造を把握
2. 並び替え案を 3 行で提示（Before / After の見出し列挙）
3. ユーザー承認後、章全体を書き換え
4. 既存の内容を **削除しない**（並び替え or 統合のみ）

**注意**：
- 章番号体系（B-2 等）は維持する（破壊的変更禁止）
- 章を分割する場合は B-2a / B-2b 等のサブ番号で（既存の章番号を空けない）

## P3：写像更新（メタ層、skill references の編集）

**該当 FB 例**：「強み 3 つの並びが応募軸でズレる」→ `interview-qa-generator/references/axis-tone.md` を更新

**手順**：
1. skill references を Read
2. FB を踏まえて該当マトリクスを更新（行追加・優先度変更）
3. 編集後、`interview-qa-generator` を実際に起動して挙動が変わることを確認
4. 旧マトリクスとの diff を `${BEHAVIOR_DOC}` のリファクタログに記録

**注意**：
- skill references の編集は本 skill が直接行う（cv-builder 等の挙動を変えるため）
- 変更後、影響を受ける企業の `想定問答_カスタム.md` の再生成提案も出す
- skill の SKILL.md 本体は基本いじらない（references のみ）

## P4：素材 + YAML + DOCX の連鎖更新

**該当 FB 例**：「副業の説明がブレる」→ 素材 42 を改訂 → YAML 両軸の experience[副業] を同期 → DOCX 再生成

**手順**：
1. 素材ファイルを **先に** 編集（事実情報の正本）
2. ユーザー承認後、`cv-builder` の SKILL.md を Read し、両軸 YAML への反映指示を案内
3. YAML 編集の前に **両軸の該当箇所** を Read で確認（軸固有の語り口だけ差を付ける）
4. `cv-builder --build all` で DOCX 再生成
5. 生成後、`cv-diff` で素材 ↔ YAML ↔ source_md の整合点検

**注意**：
- 素材は正本、YAML は表現用 → 事実は素材を真として両軸 YAML を合わせる
- 軸固有の語り口（PO 軸では「プロダクト判断経験」前面、実装軸では「実装力」前面）の差は維持

## P5：志望動機・カスタマイズ指針の改訂

**該当 FB 例**：「自社 SaaS 向けの志望動機テンプレが弱い、競合差別化が言えていない」

**手順**：
1. `${CUSTOM_PO}` or `${CUSTOM_ENG}` の業界別セクションを Read
2. 既存の強調軸・テンプレ骨子を ユーザーと一緒に見直す
3. 改訂版を直接編集
4. `${MOTIVATION_BANK}` の該当業界サンプルも同期
5. 既存応募中企業の `想定問答_カスタム.md` Q2 に「要再カスタム」マーカーを追加（`interview-qa-generator` 経由で実行）

**注意**：
- 「業界」の定義は `company-prep/references/webfetch-extraction.md` の業界判定と整合
- 既存応募中企業への波及は **企業数 × 編集量** で重くなる → 影響範囲を事前に提示

## P6：自己理解 → 連鎖リファクタ（C5 起点）

**該当 FB 例**：「PO 志向と実装志向、本当はどちらか分からない」

**手順**（順序厳守）：
1. FB を `${BEHAVIOR_DOC}` の「追加で深掘りしたい質問」に追加
2. **`deep-self-analysis` を起動** → 1 問深掘り対話を完遂
3. 対話で得た知見を `${BEHAVIOR_DOC}` の「対話で確認」章に追記（`deep-self-analysis` が実施）
4. **その知見を踏まえて** 本 skill に戻り、以下を順次更新：
   - 応募軸の方針（C4 連鎖）：`${CUSTOM_*}` の「共通方針」を見直し
   - 個別 Q の答え（C1 連鎖）：A-1 / A-6 / B-9 等の本人ベース回答を改訂
   - 自己 PR（C3 連鎖）：`${SELF_PR}` の優先順位を見直し
5. 全編集後、リファクタログに「C5 起点で C1/C3/C4 連鎖」と記録

**注意**：
- C5 を C1〜C4 と並列で進めない（深掘り前のリファクタは仮説のまま固定するリスク）
- `deep-self-analysis` 完了を待ってから次へ

## リファクタログの形式

`${BEHAVIOR_DOC}` 末尾に新セクション `## リファクタログ` を追加（無ければ作成）：

```markdown
## リファクタログ

### YYYY-MM-DD：{FB 要旨}

- **分類**：C{N} {副カテゴリ}
- **編集ファイル**：
  - {ファイル 1}（{セクション}：{要旨}）
  - {ファイル 2}（同上）
- **委譲した skill**：{cv-builder / interview-qa-generator / deep-self-analysis 等}
- **連鎖（C5 起点時）**：C5 → C{N} → C{M}
- **後続アクション**：{cv-diff drift 点検 / DOCX 再生成 / interview-qa-generator 各社実行}
```

これにより「いつ、なぜ、何を変えたか」が永続化され、半年後の再リファクタや面接振り返りで参照可能になる。

## アンチパターン（避けるべき動き）

| アンチパターン | 理由 |
|------------|------|
| 一度の起動で全カテゴリを処理 | 影響範囲が大きすぎ、変更責任が不明確 |
| ユーザー承認なしに連続編集 | 暴走、後悔の元 |
| 数値・成果を創作して素材に投入 | 24 D-1 ルール違反、面接で破綻 |
| C5 を後回しにして C1〜C4 を先行 | 仮説のままリファクタ固定、深掘り後の手戻り発生 |
| `${MASTER_QA}` の章番号体系を改訂 | 過去のログ・各社想定問答_カスタムの参照が破綻 |
| YAML を本 skill が直接編集 | cv-builder の DOCX 再生成パイプラインを通らない |
| 想定問答_カスタムを本 skill が直接編集 | 応募軸トーン再適用がスキップされる |
