# Weakness Attack Patterns — 突かれやすい箇所の検出パターン

`50_職務経歴書関連/generator/draft_*.yaml` と `40_素材/4X_職歴_*.md` を読んで、面接官が突きやすい弱み / ギャップ箇所を検出するロジック。

## 検出対象パターン

### 1. 学歴・キャリアの時系列ギャップ

| 検出条件 | 想定される深掘り質問 |
|---------|------------------|
| 大学在籍 {{ education.total_years }} 年（標準年限との差） | 「在籍期間が標準年限と異なる理由は？」「単位不足か就職活動の関係か？」 |
| 学歴ギャップ期間がある（{{ education.gap_period }}） | 「ギャップ期間中の活動内容を時系列で」「収入源と業務委託の実態」 |
| 卒業から初職就職までのギャップ | 「卒業後すぐの就職ではない理由」 |

**回答方針**：
- マスタ B-2 の 30 秒版 / 時系列版を必ず確認
- 「単位不足」は聞かれた場合のみ認める。主語は「怠慢」ではなく「確認不足から学んだ」に置く
- 商流・販売手法・過去の違和感の詳細には入らない（24 D-2 ルール）

### 2. 職種転換

| 検出条件 | 想定される深掘り質問 |
|---------|------------------|
| ネットワークエンジニア → ソフトウェアエンジニア | 「転身理由」「ネットワーク経験は活きているか」「再度ネットワークに戻る可能性」 |
| 実装 → PO / PM 志向（PO 軸時） | 「コードを書く時間が減ることへの不安は？」「技術的判断力をどう維持するか」 |
| PO / PM → 実装回帰（実装軸時） | 「マネジメントから離れる理由」「再度マネジメント候補としての評価への不安」 |

**回答方針**：
- マスタ B-1（{{ career.transition_topics[0] }}）の固定回答を流用
- 「例：Python 自動化で月間工数 8 割削減」のような実績数値は、本人が根拠を説明できる場合にのみ出す
- 24 A-6 の「技術 × 意思決定の掛け合わせが強み」を骨子に使う

### 3. 数値の検証可能性

| 検出条件 | 想定される深掘り質問 |
|---------|------------------|
| YAML / 素材に登場する % / 倍数 / 件数 | 「その数値はどう計測した？」「分母は何？」 |
| 「例：約 10 モデル実装」「例：約 20 モデルが自走」（B-4） | 「内訳と具体的なモデル名」「実装難易度の差」 |
| 「例：8 割削減」「例：60% リファクタ」 | 「比較対象は何時点と何時点？」「測定方法」 |

**回答方針**：
- 24 D-1 ルール：実測可能な数値のみ出す。曖昧な数値は「概算で」と前置きする
- 検証不能な数値は **絶対に追加で創作しない**

### 4. 副業の比重とコミット感

| 検出条件 | 想定される深掘り質問 |
|---------|------------------|
| 副業比率（実装軸 / PO 軸 YAML の experience に副業項目あり） | 「副業の週時間と本業の境界線」「本業へのコミット低下リスク」 |
| 副業案件数（{{ side_projects.projects[0].name }} / {{ side_projects.projects[1].name }} / 例：クラウドソーシング、計 {{ side_projects.total_count }} 件） | 「優先順位の付け方」「期限重複時の対応」 |
| 副業先での PO / PM 経験 | 「本業がフルスタック実装なら、副業の PO 経験を本業にどう活かす？」 |

**回答方針**：
- マスタ B-7a〜B-7d を必ず確認
- 副業比率は「平日 {{ side_projects.hours_per_week.weekday }} 時間 / 週末 {{ side_projects.hours_per_week.weekend }} 時間」を、根拠がある場合のみ実数として出せる
- 24 B-7c：「副業を辞めてほしい」要請時の回答方針を持っておく
- 企業の副業ポリシーが厳しければ Q14 で要カスタム

### 5. 個人事業の商流

| 検出条件 | 想定される深掘り質問 |
|---------|------------------|
| 個人事業の詳述（例：学習支援、例：クラウドソーシング経由） | 「販売手法」「単価」「クライアントとのトラブル経験」 |

**回答方針**：
- 24 D-2 ルール：商流・販売手法は **自分から詳述しない**
- 例示した副業領域は「そこで得た経験」に話を戻す（教育 / コミュニケーション / 国際経験など）
- 例：クラウドソーシングは「副業の入口」程度に留める、詳細は応募職種と関連が薄い

### 6. 短期離職 / 在籍期間

| 検出条件 | 想定される深掘り質問 |
|---------|------------------|
| （例：前職） 約 4 年（標準的） | 特に深掘り無し |
| （例：現職） 約 1 年で転職検討 | 「短期離職の理由」「次もすぐ辞めないか」 |
| （例：インターン先） などインターン経験 | 「正社員にならなかった理由」 |

**回答方針**：
- マスタ A-2 転職理由の「SES の階層構造で意思決定に踏み込めない」を骨子
- 「すぐ辞めない」担保として副業での継続実績（{{ side_projects.projects[0].name }} / {{ side_projects.projects[1].name }}）を出せる

### 7. ストレス・対人

| 検出条件 | 想定される深掘り質問 |
|---------|------------------|
| オフショアブリッジ SE 不在後の兼任 | 「衝突経験」「コミュニケーションコスト」 |
| クライアント常駐（例：常駐先クライアント） / 保険会社） | 「常駐先での人間関係」「複数ステークホルダー対応」 |

**回答方針**：
- マスタ Q17〜Q19（ストレス・対人質問）を準備
- 24 A-5 の弱み「自分で巻き取りすぎる傾向」と整合させる

## 検出ロジックの実装

1. YAML を読み込み、`experience[].achievements[].content` と `experience[].tasks[]` から数値（`\d+%`, `\d+割`, `\d+倍`, `約\s*\d+`）を抽出
2. 素材 `40_素材/4X_職歴_*.md` から、職歴の期間（YYYY年MM月〜）を抽出して時系列ギャップを計算
3. `meta.role` と `summary` から職種転換キーワード（"NW", "ネットワーク", "PO", "PM", "Tech Lead"）を検出
4. `experience[].employment_type` に "個人事業" / "副業" がある場合、副業項目として登録

## 出力形式

`想定問答_カスタム.md` の `## 想定される深掘り・逆襲質問` セクションに、以下の形式で追記：

```markdown
## 想定される深掘り・逆襲質問

> 本企業 / 本職種で特に突かれやすい論点。マスタ `24_面接想定問答_逆質問.md` の該当箇所と素材を併読して準備する。

### 学歴・キャリアの時系列
- 大学在籍 {{ education.total_years }} 年について（参照：24 B-2、30 秒版・時系列版の使い分け）
- {{ education.gap_reason_label }} 中の活動詳細（参照：24 B-2 深掘り想定）

### 数値の検証可能性
- 「例：約 10 モデル実装」の内訳（参照：24 B-4、Azure OpenAI / OpenAI 主要モデル名）
- 「例：Playwright 60% リファクタ」の比較対象（参照：24 B-6）

### 副業の比重
- 平日週時間と本業境界（参照：24 B-7、副業 = 平日 {{ side_projects.hours_per_week.weekday }}h / 週末 {{ side_projects.hours_per_week.weekend }}h）
- 副業ポリシー対応（参照：24 B-7c、企業の副業可否 = `${企業の副業可否}` ）

### {その他検出された論点}
- ...
```

このセクションは、面接準備時のチェックリストとして機能する。各項目で参照すべきマスタ箇所を明示し、即座に対策回答に飛べるようにする。
