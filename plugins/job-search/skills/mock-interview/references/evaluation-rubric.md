# Evaluation Rubric — 5 軸評価ルーブリック

模擬面接終了後、ユーザーの回答を 5 軸で評価する判定基準。各軸 ◎○△✕ + 根拠コメント 1 行以上。

## 5 軸定義

### 軸 1：説得力

**問**：主張に裏付けがあるか（具体例・数値・経験）

| 評価 | 判定基準 |
|-----|---------|
| ◎ | 主張 + 具体例 + 数値 + 第三者検証可能な事実 すべて揃う |
| ○ | 主張 + 具体例（数値はあれば望ましい）|
| △ | 主張のみ、具体例が抽象的 / 一般論 |
| ✕ | 主張が曖昧、または聞かれたことに答えていない |

**例**：「AI に強いです」
- ✕：抽象的すぎ、何を以て強いか不明
- △：「現職で AI を担当しています」（具体例だが薄い）
- ○：「例：現職で Azure OpenAI の主担当として約 10 モデル実装しました」
- ◎：「例：現職で Azure OpenAI の主担当として、複数モデルを一定期間で実装し、SDK 移行も含めてテンプレ化しました。実装後はチームに委譲し、追加モデルが自走で増えています」

### 軸 2：具体性

**問**：抽象論で逃げず、固有名詞・期間・規模を出せたか

| 評価 | 判定基準 |
|-----|---------|
| ◎ | 固有名詞（製品名 / ライブラリ / モデル名）+ 期間 + 規模 すべて出る |
| ○ | 上記 2 つ以上 |
| △ | 1 つだけ、または曖昧（「数ヶ月」「複数」等）|
| ✕ | 全部抽象（「色々」「いっぱい」「効率的に」）|

**減点ポイント**：
- 「効率的に」「適切に」「臨機応変に」等の中身のない副詞
- 「いつもこうしてます」（再現性不明）

### 軸 3：深掘り耐性

**問**：突っ込まれた時に崩れず、追加情報を出せたか

| 評価 | 判定基準 |
|-----|---------|
| ◎ | 3 段階深掘りされても具体性が落ちず、最後は「失敗経験 / 学び / 今ならどう変えるか」まで自分から出せる |
| ○ | 2 段階までは具体的、3 段階目で少し弱まるが論理破綻はしない |
| △ | 1 段階目で抽象化、2 段階目で「ちょっと覚えてない」 |
| ✕ | 1 段階目で詰まる、または取り繕いで矛盾発生 |

**特別判定**：
- 「知らない」を明言できる → △ 以上保証（言い訳より高評価）
- 「知らないが、類推するなら……」と仮説提示 → ○ 以上
- 知ったかぶり → ✕ 確定

### 軸 4：応募軸整合

**問**：強調軸が応募軸（PO / 実装）と一致しているか、トーンダウンすべき箇所を回避できたか

| 評価 | 判定基準 |
|-----|---------|
| ◎ | 応募軸の強調軸 3 つを的確に使い、トーンダウンすべき領域（PO 軸なら「コードが好き」、実装軸なら「マネジメント志向」）を回避 |
| ○ | 強調軸はおおむね合うが、1〜2 箇所トーンダウン領域に踏み込む |
| △ | 強調軸が散漫、応募軸の核心と関係ない話が長い |
| ✕ | 応募軸と真逆の自己 PR（PO 軸面接で「コード書きたいです」を前面に等）|

参照：`interview-qa-generator/references/axis-tone.md` の強調 / トーンダウンマトリクス

### 軸 5：マナー

**問**：24 D 章の NG ルールを踏んでいないか

| 評価 | 判定基準 |
|-----|---------|
| ◎ | NG ルール全 6 項目を完全回避 + 面接官への礼節 |
| ○ | NG を 1 件踏むが軽微（聞かれて答えた範囲）|
| △ | NG を 2 件以上 / または聞かれていないのに自分から踏む |
| ✕ | 検証不能な数値を即興で創作 / 家族事情を年収根拠 / 個人事業の商流詳述 |

**NG ルール再掲（24 D 章）**：
- D-1：検証不能な成果数値の創作
- D-2：個人事業の商流・販売手法を自分から詳述
- D-3：家族 / 生活事情を年収根拠にする
- D-4：他社批判
- D-5：エージェントの個別名を出して比較する
- D-6：副業先の機密情報を漏らす

## 総合評価の算出

各軸 ◎=4 / ○=3 / △=2 / ✕=1 として平均：

| 平均 | 総合 | 本番想定 |
|-----|-----|-------|
| 3.5 以上 | ◎ | 通過しそう |
| 2.7〜3.4 | ○ | 五分（補強次第） |
| 2.0〜2.6 | △ | 厳しい（要対策） |
| 2.0 未満 | ✕ | 大幅な見直し必要 |

ただし **マナー軸が ✕ なら総合は最大 △**（マナー違反は本番で即落選リスク）。

## 質問ごとの良し悪し記載フォーマット

```markdown
### Q{N}. {質問}

回答：
> {ユーザー回答の要約 or 全文}

深掘り（あれば）：
> Q: {深掘り}
> A: {ユーザー回答}

評価：
- 説得力: ○ / 具体性: △ / 深掘り耐性: ○ / 軸整合: ◎ / マナー: ◎
- ✅ 良かった点：応募軸（PO）の強調軸 3 つを順序通り出せた
- 🔧 改善余地：「例：約 10 モデル」の内訳が抽象的、具体的なモデル名 3 つは即答できると ◎
- 📌 参照：24 B-4 / 41_職歴_（例：現職）.md の生成 AI セクション
```

## 改善優先度の決め方

評価終了後、改善余地を以下の優先度でソートして「上位 3 件」を提示：

1. **マナー違反**（最優先、即対策）
2. **応募軸ズレ**（複数 Q で同じ方向にズレ → 軸再設定の可能性）
3. **深掘り耐性 ✕ の論点**（同論点が複数 Q で出るなら準備不足）
4. **説得力 ✕ / △ で頻出する論点**（数値曖昧・固有名詞欠如）
5. **具体性 △ で頻出する論点**

優先度 1〜2 → `feedback-integrator` 起動推奨
優先度 3〜5 → `interview-qa-generator --section deep-attack` 起動推奨

## 評価コメントの書き方

### Do

- 具体的に：「Q4 で副業の説明がブレた」
- 参照を添える：「24 B-7 の固定回答を再確認」
- 次の練習を示唆：「次は pinpoint mode で B-7 集中」

### Don't

- 抽象的：「もう少し頑張りましょう」
- 過度な励まし：「素晴らしい！」（根拠なし）
- 過度な厳しさ：「不合格です」（模擬は学習目的）
- 二値裁定：「正解 / 不正解」
